01 // ARCHITEKTURA SYSTEMU

Fraction Guild Clans v2.0 to samowystarczalny plugin na serwery Paper/Spigot/Purpur (Minecraft 1.20–1.21.11). Nie wymaga zewnętrznych zależności. Cała logika rozgrywkowa, ekonomiczna i ochronna jest definiowana deklaratywnie poprzez pliki konfiguracyjne YAML.

PAPER API
1.20.x – 1.21.11
CORE MODULES
guild, economy, lang, database
OPTIONAL MODULES
cuboid, egg, ranking, gui, tab...
PERSISTENCE
SQLite / MySQL / PostgreSQL

Moduły ładują się zgodnie z zależnościami (sortowanie topologiczne). Wyłączenie modułu zwalnia jego listenery, komendy i taski. Stack: Java 17+, Maven/Gradle, HikariCP, MiniMessage + legacy, własny GUI oparty na Bukkit Inventory API.

02 // KOMENDY SYSTEMOWE

Szybkie referencje dla administratora serwera i dewelopera.

# Podstawowe komendy gracza
/guild create <nazwa> <tag>
/guild invite <gracz> / /guild join <gildia>
/guild menu / /guild top / /guild map
# Komendy administratora
/guild admin module list/enable/disable/reload [moduł]
/guild admin backup / /guild admin restore <data>
/guild admin lang reload / /guild admin debug

04 // FILOZOFIA I WYMAGANIA

FGC to zaawansowany plugin serwerowy przeznaczony dla Paper, Spigot i Purpur, łączący klasyczne systemy gildii z nowoczesną architekturą modularną. Filozofia "Everything is Configurable" oznacza, że każdą funkcjonalność można włączyć lub wyłączyć, każdy koszt i wiadomość są konfigurowalne, a każdy element GUI można zmienić bez znajomości programowania.

  • Zero zależności: nie wymaga Vault, WorldGuard, WorldEdit, EssentialsX ani CMI.
  • Opcjonalne integracje: PlaceholderAPI (placeholdery %fgc_*), EcoCore (zaawansowana ekonomia).
  • Wspierane silniki: Paper 1.20.x–1.21.11 (zalecany), Spigot 1.20.x–1.21.11, Purpur 1.20.x–1.21.11.
  • Filozofia modułowa: każda funkcja to osobny plik YAML w modules/, aktywowany jednym przełącznikiem enabled.

05 // INSTALACJA I STRUKTURA PLIKÓW

Instalacja sprowadza się do umieszczenia pliku FractionGuildClans-2.0.jar w katalogu plugins/ serwera. Po restarcie plugin automatycznie generuje strukturę katalogów i plików konfiguracyjnych z komentarzami.

Katalog/PlikPrzeznaczenie
config.ymlGłówna konfiguracja: parametry globalne, prefixy, formaty wiadomości, TPS
lang/Pliki językowe (pl_PL.yml, en_US.yml, de_DE.yml...) z hot-reloadem
modules/14 plików YAML, każdy odpowiada za jeden moduł funkcjonalny
data/Baza danych: SQLite domyślnie, opcja MySQL/PostgreSQL w config.yml
backups/Automatyczne kopie zapasowe ZIP z datownikiem
logs/Logi transakcji ekonomicznych, zmian w gildiach, audyt bezpieczeństwa

06 // ECOCORE -- WŁASNY SYSTEM EKONOMII

FGC v2.0 nie używa Vault. Zamiast tego oferuje EcoCore – autorski system ekonomiczny tego samego autora, lub tryb standalone oparty na itemach.

  • EcoCore: wielowalutowość, banki gildii z odsetkami, transakcje międzygildiowe, pełny audyt.
  • Tryb standalone: itemy (domyślnie diamenty) jako waluta; komendy /guild deposit, /guild withdraw działają identycznie.
  • Przełączanie: parametr mode: "ecocore" lub mode: "items" w modules/economy.yml + reload modułu.
# Przykład konfiguracji waluty w economy.yml
currencies:
primary: { name: "Moneta", symbol: "&e", format: "{amount} {symbol}", decimals: 2 }
bank: { enabled: true, interest_rate: 0.5, interest_interval: "24h" }

07 // CUBOIDY I WŁASNA OCHRONA

Wbudowany system ochrony oparty jest na cuboidach (obszarach cylindrycznych) wokół jaja gildii. Nie wymaga WorldGuard ani WorldEdit, co eliminuje konflikty priorytetów regionów i zmniejsza narzut wydajnościowy.

  • 3 strefy: spawn (10% promienia, tylko lider/co-lider), build (75%, członkowie + zaufani), PvP (100%, wszyscy mogą walczyć).
  • 7 flag: build, destroy, pvp, use, bucket, entry, interact – wartości ALLOW/DENY/MEMBERS/ALLIES/LEADER.
  • Trusted Players: zaufani gracze spoza gildii z indywidualnymi uprawnieniami, domyślnie limit 10 per gildia.
  • System levele: 10 poziomów cuboidu, zwiększających promień; wymagania: czas gildii, punkty, członkowie, itemy/waluta.

08 // SYSTEM PLIKÓW JĘZYKOWYCH

System i18n oparty na plikach YAML w katalogu lang/. Wspiera hierarchiczne klucze, MiniMessage, legacy &k, placeholdery oraz hot-reload bez restartu.

  • Fallback: aktywny język → en_US.yml → raw key. Nigdy nie zwraca pustej wiadomości.
  • Formatowanie: MiniMessage (<color>, <bold>, <gradient>) oraz legacy (&a, &l) rozpoznawane automatycznie.
  • Placeholdery: {player}, {guild}, {tag}, {points}, {level}, {lives}, {radius}, {members}, {balance}, {time}, {sender}, {target}, {amount}, {item}, {price}, {rank}, {position}.
  • Kategorie prefiksów: prefix.info, prefix.error, prefix.success, prefix.warning.
# Hot-reload języka
/guild admin lang reload

09 // SYSTEM GILDII

Pełny system zarządzania gildią: zakładanie, członkowie, rangi, relacje i rozwiazanie.

  • Zakładanie: /guild create <nazwa> <tag> [kolor] – walidacja nazwy, unikalności, itemów, odległości od innych gildii.
  • Rangi: LEADER, CO_LEADER, MODERATOR, MEMBER, RECRUIT – z konfigurowalnym zestawem uprawnień w modules/guild.yml.
  • Członkowie: zaproszenia, join, leave, kick, ban/unban, promote/demote, limity zależne od poziomu.
  • Ustawienia: sethome/home z opóźnieniem, opis, kolor, flagi gildii (public, allow-join-requests z systemem próśb, show-home).
  • Relacje: sojusze (/guild ally), wojny (/guild enemy), neutral, truce – wpływają na PvP i dostęp do terenów.
  • Rozwiazanie: potwierdzenie, opcjonalny zwrot kosztów, czyszczenie danych, broadcast serwerowy.

10 // JAJO GILDII -- SYSTEM OBRONY

Jajo gildii to serce systemu obrony. Jego zniszczenie prowadzi do upadku gildii po czasie karencji.

  • 3 etapy zniszczenia: Pełne (3/3) → Uszkodzone (2/3) → Krytyczne (1/3) → Zniszczone (0/3).
  • Efekty: cząsteczki, dym, dźwięki, ciemność i słabość dla członków, broadcast serwerowy.
  • Ochrona: tarcza startowa 30 min, komenda /guild egg shield (płatna), /guild egg shield.
  • Naprawa: /guild egg repair przywraca 1 życie; cooldown 24h globalnie, wymagane itemy.
  • Upadek: po zniszczeniu 60 minut na obronę; broadcast, Hall of Shame, nagroda dla niszczycieli.

11 // WIEŚNIACY HANDLOWI

Customowe NPC gildii oferują handel, zadania i wymianę surowców.

  • Tworzenie: /guild villager create <typ> <nazwa> – tylko w obrębie własnego cuboidu, limit zależny od poziomu.
  • Typy: Handlarz, Zleceniodawca, Straznik (kowal, alchemik, bibliotekarz, rybak, łowca).
  • Handel: oferty konfigurowane w YAML, GUI wymiany, cooldown transakcji globalny, poziom relacji wpływa na ceny.
  • Zadania dzienne: zabij, zbierz, obron – z nagrodami i systemem reputacji.
  • Ochrona: interakcja tylko dla członków gildii; NPC są invulnerable.

12 // GUI -- INTERFEJS GRAFICZNY

Pełne menu graficzne konfigurowane z plików YAML. Wszystkie ekrany wspierają animacje, dźwięki i dynamiczne odświeżanie.

  • Główne menu (/guild menu): info, członkowie, cuboid, ranking, bank, jajo, sojusze.
  • Podmenu: zarządzanie członkami, ustawienia gildii (toggle, opis, kolory, flagi), handel z wiesniakiem, ranking.
  • Animacje: fade_in, slide_up, random_fill; dźwięki przy otwarciu, kliknięciu i zamknięciu.
  • Dynamic refresh: aktualizacja slotów co X ticków bez zamykania GUI; lazy refresh oparty na hashu zawartości.

13 // RANKING GILDII

System rankingowy oparty na algorytmie ELO z asystami, K/D/A i sezonami.

  • ELO: base_k: 32, k_scaling, start_points: 1000, min_points: 1, szybki tryb rankingowy (pierwsze 10 gier).
  • Asysty: time_window 15s, min 15% obrażeń, max 3 asysty za zabójstwo.
  • Topki: /guild top (gildie), /guild top players (gracze) z paginacją i filtrowaniem.
  • Sezony: konfigurowalny czas (domyślnie 3 miesiące), opcjonalny reset punktów, finale multiplier, nagrody TOP 3.
  • Hall of Fame: archiwum zwycięzców poprzednich sezonów dostępne w grze i na stronie HTML.

14 // MAPA SERWERA HTML

Interaktywna mapa gildii generowana do przeglądarki oraz uproszczona mapa ASCII w czacie.

  • HTML: statyczny HTML+CSS+JS w folderze web/map/, render chunków do PNG, kolorowe poligony cuboidów, markery jaj.
  • Aktualizacja na żywo: tryb polling (domyślnie), WebSocket lub none – zależnie od config.yml.
  • Panel boczny: lista gildii; kliknięcie centruje mapę i pokazuje informacje.
  • ASCII: /guild map – symbole # (twoja), - (wroga), + (sojusznicza), kompas N/E/S/W.

15 // CUSTOM TABULATOR

Niestandardowa lista graczy (TAB) z rankingiem, prefixami gildii i animacjami.

  • 3 kolumny: lewa (TOP 20 graczy), środkowa (gracze online), prawa (TOP 10 gildii).
  • Nagłówek/stopka: nazwa serwera, sezon, gracze online, TPS, RAM, adres strony.
  • Sortowanie: gildia → punkty → alfabetycznie; formatowanie nicków z tagiem gildii.
  • Wsparcie PlaceholderAPI: %guild_name%, %guild_tag%, %guild_rank%, %guild_points%, %guild_level%.

16 // SYSTEM MODUŁÓW I KONFIGURACJA

Architektura modułowa to fundament wersji 2.0. Każdy moduł to samodzielna jednostka z własnymi komendami, listenerami i plikiem YAML.

  • Cykl życia: DISABLED → LOADING → ENABLED → RUNNING. Wyłączenie zwalnia wszystkie zasoby.
  • Zależności: sortowanie topologiczne podczas startu; brak wymaganego modułu = błąd w logach i pozostawienie w stanie nieaktywnym.
  • ConfigManager: wczytywanie i walidacja YAML, auto-generowanie domyślnych configów, migracja schematu.
  • DatabaseManager: abstrakcja SQLite/MySQL/PostgreSQL, HikariCP, automatyczne migracje schematu.
  • Webhooki i backup: powiadomienia Discord o założeniu/upadku/sojuszach/wojnach; kopie zapasowe ZIP z retencją i przywracaniem.
